# Contract: Paperclip Container Image

**Built by**: `.github/workflows/build-image.yml`
**Stored at**: Artifact Registry, `us-central1-docker.pkg.dev/paperclip-492823/paperclip/paperclip:<digest>`
**Consumed by**: Cloud Run service (`infra/modules/compute/`) and Cloud Run Job (`infra/modules/jobs/`)
**Source**: **Upstream Paperclip's official `Dockerfile`** at `https://github.com/paperclipai/paperclip/blob/master/Dockerfile`, built at the pinned `paperclip_version` tag from `infra/envs/prod/versions.tfvars`. **We do NOT maintain a custom Dockerfile.**

## Why we inherit upstream's Dockerfile

Paperclip's upstream Dockerfile is non-trivial. It builds three workspaces in a specific order (`@paperclipai/ui` → `@paperclipai/plugin-sdk` → `@paperclipai/server`), installs the `@anthropic-ai/claude-code`, `@openai/codex`, and `opencode-ai` CLIs globally so the spawned-subprocess agents work, runs as the `node` user with configurable UID/GID via build args, and ships an entrypoint that remaps the user to match the volume mount. Inheriting their Dockerfile means we get all of that for free, and we automatically pick up upstream improvements when we bump the pinned `paperclip_version`.

The only thing our CI workflow does differently from a vanilla `docker build .` is **clone the upstream repo at the pinned tag** before building. The build context is the upstream tree, not our deployment-spec repo.

## Build inputs

| Input | Source | Notes |
|---|---|---|
| Paperclip release tag | `infra/envs/prod/versions.tfvars` (`paperclip_version`) | Pinned for the single environment. CI clones `paperclipai/paperclip` at this exact tag. |
| Docker build context | The cloned upstream tree | NOT this repo. CI does `cd paperclip-source && docker build .`. |
| `USER_UID` / `USER_GID` build args | Optional, default 1000 | Upstream's Dockerfile accepts these for matching host UIDs in dev. We don't override them in CI — Cloud Run doesn't use volume mounts, so the default UID/GID is fine. |
| Build-time secrets | None | The image MUST contain zero secret material. The build context never receives secrets, and upstream's Dockerfile doesn't pull any. |

## Runtime port

**3100** — this is Paperclip's default and what their official Dockerfile `EXPOSE`s. The Cloud Run service spec sets `containerPort: 3100`; Cloud Run injects `PORT=3100` at task launch.

## Runtime user

`node` — the standard Node base image user, UID 1000 by default. Upstream's `docker-entrypoint.sh` calls `gosu node` to drop privileges before exec'ing the server.

## Runtime inputs (env vars, set by Cloud Run at deploy time)

The compute module's Cloud Run service spec sets the following env vars on the `paperclip` service. Variables in **bold** are required; others are optional with documented defaults.

### Required from Secret Manager (mounted via Cloud Run `env.value_source.secret_key_ref`)

| Cloud Run env var | Secret Manager secret | Purpose |
|---|---|---|
| **`PAPERCLIP_SECRETS_MASTER_KEY`** | `paperclip-master-key` | 32-byte master key for Paperclip's local-encrypted secret store. Read directly from the env var by `server/src/secrets/local-encrypted-provider.ts` (preferred over the file-path variant `PAPERCLIP_SECRETS_MASTER_KEY_FILE`). Generated by `bootstrap-master-key.sh`. |
| **`BETTER_AUTH_SECRET`** | `paperclip-better-auth-secret` | 32-byte signing secret for Better Auth (cookies/JWTs) and the agent JWT system (`PAPERCLIP_AGENT_JWT_SECRET` falls back to this same value, so one secret covers both). REQUIRED for any `PAPERCLIP_DEPLOYMENT_MODE=authenticated` deployment — Paperclip refuses to start without it. Generated by `bootstrap-better-auth-secret.sh`. |
| **`DATABASE_URL`** | `paperclip-database-url` | `postgres://paperclip:<pw>@<private-ip>/paperclip` connection string for Cloud SQL via Serverless VPC Connector. Created by the database module in Phase 3. |
| **`AWS_ACCESS_KEY_ID`** | `paperclip-s3-access-key-id` | GCS HMAC interop access ID. Paperclip's S3 storage provider creates an `S3Client` without explicit credentials, so the AWS SDK reads `AWS_ACCESS_KEY_ID` / `AWS_SECRET_ACCESS_KEY` from the env (server/src/storage/s3-provider.ts). The Secret Manager secret is named after its content (`paperclip-s3-*`), but the env var must be the standard AWS SDK name. |
| **`AWS_SECRET_ACCESS_KEY`** | `paperclip-s3-secret-access-key` | GCS HMAC interop secret. Same rationale as `AWS_ACCESS_KEY_ID`. |

**No `ANTHROPIC_API_KEY` and no `OPENAI_API_KEY` are set.** Vertex Claude uses the Cloud Run service account's `roles/aiplatform.user` permission. Verified end-to-end on 2026-04-10 (Phase B).

| `SERVE_UI` | `true` | Required to serve the React dashboard alongside the REST API. Without this Paperclip only exposes the API. |
| `PAPERCLIP_HOME` | `/paperclip` | Matches upstream Dockerfile's `HOME` and `PAPERCLIP_HOME` defaults. The directory exists in the image (`mkdir -p /paperclip` in upstream's production stage). |

| **`PAPERCLIP_DEPLOYMENT_MODE`** | `authenticated` | First half of the public-mode config. Allowed values per `packages/shared/src/constants.ts`: `local_trusted` \| `authenticated`. |
| **`PAPERCLIP_DEPLOYMENT_EXPOSURE`** | `public` | Second half of the public-mode config. Allowed values: `private` \| `public`. The two-var pair makes this an internet-facing authenticated deployment. |
| **`PAPERCLIP_PUBLIC_URL`** | `https://<your-domain>` | Explicit public URL required by `authenticated+public` mode. Better Auth uses this as the canonical issuer URL for sessions and as the base for cookie scoping. The exact hostname is set in `terraform.tfvars`. |

| `PAPERCLIP_STORAGE_PROVIDER` | `s3` | Switches Paperclip from `local_disk` (default) to its S3-compatible storage backend. |
| `PAPERCLIP_STORAGE_S3_BUCKET` | `paperclip-492823-uploads` | The deployment's GCS bucket name (S3-compatible interop). |

| `PAPERCLIP_STORAGE_S3_REGION` | `us-central1` | The bucket's region. The AWS SDK requires a region; GCS interop accepts the GCP region name. |
| `PAPERCLIP_STORAGE_S3_FORCE_PATH_STYLE` | `true` | GCS interop typically requires path-style addressing (vs. virtual-host style). |

## Boot contract (upstream `scripts/docker-entrypoint.sh`)

Upstream's entrypoint is minimal — it remaps the `node` user UID/GID to match the volume mount (a no-op on Cloud Run since we don't use volume mounts), then `exec gosu node "$@"` to drop privileges and run the CMD.

The actual server start command (from upstream's CMD) is:

```
node --import ./server/node_modules/tsx/dist/loader.mjs server/dist/index.js
```

We don't override it. The compute module's `command` / `args` fields are left unset so Cloud Run uses the image's CMD.

## Cloud Run Job mode (paperclipai-doctor and paperclipai-bootstrap-ceo)

The same image, when launched as a Cloud Run Job, runs Paperclip CLI commands instead of the server. The job overrides the entrypoint command:

**For doctor**:
```
command: ["pnpm", "paperclipai", "doctor"]
```

**For bootstrap-ceo (one-time first-admin invite)**:
```
command: ["pnpm", "paperclipai", "auth", "bootstrap-ceo"]
args:    ["--base-url", "https://<deployment-domain>"]
```

The `paperclipai` CLI is the script entry from upstream's `package.json` (`node cli/node_modules/tsx/dist/cli.mjs cli/src/index.ts`), invoked via `pnpm` so the workspace context is set correctly.

Doctor MUST exit non-zero on any failure so the calling `gcloud run jobs execute --wait` reports failure to the deploy script.
Bootstrap-ceo prints the one-time invite URL to stdout; the deploy procedure captures it from the Cloud Run Job execution log and surfaces it to the operator.

## Image hygiene

Inherited from upstream:

- Multi-stage build (base → deps → build → production)
- Production stage strips dev dependencies
- Runs as non-root `node` user (uid 1000 by default)
- No package managers in the production stage beyond what upstream installs (pnpm via corepack, plus the npm-installed Claude Code / Codex / OpenCode CLIs)
- No secrets, no `.env` files, no service account JSON keys, ever

What our CI adds on top:

- **Trivy scan** in CI must pass HIGH/CRITICAL with no unfixed findings
- We do NOT run Hadolint because we don't author the Dockerfile

## Versioning

The image is identified to the deployment **only** by its immutable digest (`sha256:…`). Tags exist for human inspection (`<paperclip-version>`, `<git-sha>`) but are never referenced by the Terraform service spec.
